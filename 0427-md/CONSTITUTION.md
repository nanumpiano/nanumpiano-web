# CONSTITUTION.md

- 문서 상태: 프로젝트 재개용 최상위 헌법
- 적용 범위: 나눔프로젝트 계열 전체, 현재는 나눔피아노에 우선 적용
- 작성 기준일: 2026-04-27
- 작성자: ChatGPT / 사용자 검토 필요
- 문서 성격: 설명문이 아니라 작업 재개용 복구 문서

---

## 0. 이 문서의 목적

이 문서는 특정 폴더 하나에 묶인 안내문이 아니다.  
작업자가 바뀌거나, AI 세션 기억이 붕괴되거나, 빈 폴더에서 프로젝트를 다시 이어받아야 할 때 **가장 빠르게 원상복구하기 위한 최상위 복구 문서**다.

이 문서를 읽으면 즉시 아래 다섯 가지가 복구되어야 한다.

1. 프로젝트가 무엇인지
2. 지금 어디까지 왔는지
3. 무엇이 확정되었는지
4. 지금 당장 무엇을 해야 하는지
5. 무엇을 하지 말아야 하는지

## 1. 문서 체계 원칙

### 1-1. 기본 문서 3개

루트에서 작업자가 반드시 먼저 읽어야 하는 기본 문서는 아래 3개다.

```text
CONSTITUTION.md
WORK_LOG.md
나눔피아노_전체_설계.md
```

각 역할은 아래로 고정한다.

| 문서 | 역할 |
|---|---|
| `CONSTITUTION.md` | 최상위 헌법, 작업 재개 절차, 금지 사항, 도구 운용 원칙 |
| `WORK_LOG.md` | 최신 작업이 맨 위에 오는 역순 작업 블랙박스 |
| `나눔피아노_전체_설계.md` | 나눔피아노 작동 원리, 제품 구조, 화면 구조, 개발 개선 토탈 설계도 |

### 1-2. 부수 문서 허용

개발 중 필요한 부수 문서는 허용한다.  
단, 종류를 최대한 적게 유지한다.

허용되는 부수 문서 범주는 아래뿐이다.

```text
/adapters/codex/        Codex 전용 설치·운용 문서
/adapters/cursor/       Cursor 전용 설치·운용 문서
/adapters/antigravity/  Antigravity 전용 하네스·운용 문서
/work/                  임시 작업 계획, 임시 조사
/references/            외부 자료 요약
/evidence/              스크린샷, 로그, 비교 증거
/archive/               더 이상 정본이 아닌 이전 문서 보관
/tools/                 스크립트, 설정 예시
```

부수 문서는 최종 판단 기준이 아니다.  
부수 문서의 확정 내용은 반드시 기본 문서 3개 중 하나에 흡수한다.

### 1-3. 문서 남발 금지

아래 행위는 금지한다.

- 루트에 새 `.md` 문서를 계속 만드는 행위
- 같은 내용을 여러 문서에 반복하는 행위
- `PRINCIPLES.md`, `STRUCTURE.md`, `SOURCES.md`, `AI.md`처럼 역할이 겹치는 문서를 다시 루트 정본으로 만드는 행위
- 임시 조사 문서를 정본처럼 남기는 행위
- 이전 문서를 삭제하면서 흡수 기록을 남기지 않는 행위

문서가 필요해 보이면 먼저 아래 질문에 답한다.

1. 작업 기록인가? → `WORK_LOG.md`
2. 작동 원리나 제품 설계인가? → `나눔피아노_전체_설계.md`
3. 작업 규칙, 금지, 도구 운용 원칙인가? → `CONSTITUTION.md`
4. 특정 도구 설치·설정인가? → `/adapters/<tool>/`
5. 일회성 조사인가? → `/work/` 또는 `/references/`, 이후 3대 문서로 흡수

## 2. 최상위 원칙

### 2-1. 거짓 금지

확인하지 않은 내용을 사실처럼 쓰지 않는다.  
모르면 모른다고 적는다.  
추정은 반드시 “추정”이라고 표시한다.

사용자가 직접 제공한 실행 화면, 설명문, 직접 수정한 문장, 작업 순서, 첨부 문서와 이미지는 최상위 사실 자료다.  
소스코드만 보고 실제 작동 원리를 단정하지 않는다.

### 2-2. 작동 원리 우선

구현 전 반드시 작동 원리를 먼저 고정한다.

특히 나눔피아노에서는 아래 구분을 절대 섞지 않는다.

- Step 1
- Step 2
- Editor
- 저장
- 재적용
- AI 코칭

이 중 무엇을 수정하는지 답하지 못하면 작업하지 않는다.

### 2-3. 현재 상태 확인 우선

작업자는 먼저 아래를 확인한다.

1. `CONSTITUTION.md`
2. `WORK_LOG.md`
3. `나눔피아노_전체_설계.md`
4. 현재 파일 목록
5. 현재 Git 상태
6. 현재 실행 가능 여부
7. 마지막 검증 결과
8. 마지막 실패와 남은 문제

### 2-4. 계획 우선, 단 절차 과잉 금지

작업이 작고 명확하면 간단 계획으로 충분하다.  
작업이 구조 변경, 대규모 수정, 삭제, 이동, 배포, 도구 설정이면 반드시 계획서를 먼저 보고한다.

계획은 삽질 방지를 위한 장치이지, 작업을 느리게 만들기 위한 의식 절차가 아니다.

### 2-5. 검증 없는 완료 금지

아래 중 하나 이상이 없으면 완료라고 말하지 않는다.

- diff
- 실행 로그
- 테스트 결과
- 브라우저 검수 결과
- 스크린샷
- 파일 목록 비교
- 사용자 확인
- Git 커밋 해시

### 2-6. WORK_LOG 삭제 금지

`WORK_LOG.md`는 블랙박스다.

금지 사항:

- 이전 날짜 기록 삭제
- 실패 기록 삭제
- 작업 내용을 요약해 덮어쓰기
- 전날 기록을 정리한다는 명목으로 축약
- 최신 기록을 맨 아래에 붙이는 방식
- 작업자와 시간을 누락하는 방식

### 2-7. 최신 기록은 맨 위

`WORK_LOG.md`는 **역순**으로 쓴다.  
가장 최근 작업이 항상 맨 위에 있어야 한다.

각 항목 제목에는 반드시 날짜, 시간, 작업자를 쓴다.

예:

```md
## 2026-04-27 18:40 / 작업자: Codex / 작업: 문서 통합 조사
```

### 2-8. 작업자 교체 대비

작업자는 언제든 바뀔 수 있다.  
따라서 작업자는 “내가 기억한다”는 전제로 일하지 않는다.

모든 결정, 실패, 검증, 남은 문제는 문서에 남긴다.  
다음 작업자가 `CONSTITUTION.md`, `WORK_LOG.md`, `나눔피아노_전체_설계.md`만 읽고 바로 이어받을 수 있어야 한다.

## 3. 나눔피아노 프로젝트 현재 고정 원리

나눔피아노는 **Editor에서 악보를 만들거나 수정하고, 저장된 악보를 Step 1에서 한 음씩 익히고, Step 2에서 속도에 맞춰 흐름 훈련하며, 부족한 부분은 다시 Editor로 수정해 Step 1과 Step 2에 재적용하는 피아노 학습 순환 시스템**이다.

AI 코칭은 이 흐름을 대체하지 않는다.  
AI 코칭은 사용자가 지금 무엇을 해야 하는지 설명하고, 난도를 낮추며, 다음 행동을 안내하는 보조 계층이다.

## 4. 현재 프로젝트 복구 질문

새 작업자는 작업 전에 아래 질문에 답해야 한다.

### 4-1. 프로젝트가 무엇인가?

나눔피아노는 악보 생성, 학습, 수정, 재학습이 이어지는 피아노 학습 시스템이다.

### 4-2. 지금 어디까지 왔는가?

현재까지 확인된 범위에서는 1차 전시용 웹 프리뷰가 구성되어 있다.

핵심 프리뷰 흐름은 아래다.

```text
index.html
preview/step1_v3.html
preview/step2.html
preview/editor_flow.html
preview/reapply_flow.html
```

이 파일들은 실제 제품 완성본이 아니라, 작동 원리와 전시용 흐름을 보여주는 웹 프리뷰다.

### 4-3. 무엇이 확정되었는가?

- Step 1은 한 음씩 따라치는 즉시 학습 단계다.
- Step 2는 같은 악보를 템포 조절해 흐름 훈련하는 단계다.
- Editor는 실제 악보를 만들고 수정하는 생산 도구다.
- 저장은 끝이 아니라 재학습으로 돌아가기 위한 중간 단계다.
- 재적용은 저장된 악보가 다시 Step 1과 Step 2에서 작동하는 순환이다.
- AI 코칭은 전체 흐름 위의 안내·분석 계층이다.
- 작업 문서는 3개 중심으로 통합한다.
- WORK_LOG는 최신순으로 쓰고 삭제하지 않는다.

### 4-4. 지금 당장 무엇을 해야 하는가?

상황별로 다르지만 기본 순서는 아래다.

1. 3대 문서를 읽는다.
2. 현재 폴더 구조를 확인한다.
3. 현재 Git 상태를 확인한다.
4. 마지막 WORK_LOG 맨 위 항목을 읽는다.
5. 현재 작업 범위를 사용자에게 재확인한다.
6. 수정 전 계획 또는 조사 보고를 한다.
7. 승인 후 수정한다.
8. diff와 검증 결과를 보고한다.
9. WORK_LOG 맨 위에 새 기록을 추가한다.

### 4-5. 무엇을 하지 말아야 하는가?

- 작동 원리 이해 없이 구현하지 않는다.
- Step 2를 속도 설명 카드로 만들지 않는다.
- Editor를 단순 장난감 UI로 만들지 않는다.
- 재적용을 저장 후 데이터 처리 다이어그램으로만 표현하지 않는다.
- 기존 WORK_LOG를 정리한다며 삭제하지 않는다.
- 루트에 문서를 계속 늘리지 않는다.
- Codex, Cursor, Antigravity에게 무제한 권한을 주고 방치하지 않는다.
- 사용자 승인 전 삭제, 이동, 대규모 구조 변경, 커밋, 배포를 하지 않는다.

## 5. 하네스와 도구 운용 원칙

### 5-1. 자동으로 적용되는 영역

도구별로 자동 또는 반자동으로 참고되는 설정이 있다.

예:

- `.antigravityignore`
- `agent/rules`
- `agent/skills`
- `agent/workflows`
- `AGENTS.md`
- `.cursor/rules`
- `.codex/config.toml`
- MCP 서버 설정

이 설정들은 작업자를 돕는 장치다.  
그러나 자동 적용을 맹신하지 않는다.  
작업 시작 시 반드시 “규칙을 읽었는지” 확인하게 한다.

### 5-2. 사용자가 전략적으로 지시해야 하는 영역

아래는 자동으로 믿지 말고 사용자가 직접 지시해야 한다.

- 모델 선택
- 계획 모드 사용 여부
- 브라우저 검수 여부
- MCP 사용 여부
- Full Access 또는 고위험 권한 사용 여부
- 배포 여부
- 삭제 또는 이동 여부

### 5-3. 하이브리드 운영 원칙

고정 시스템과 유연한 도구 운용을 섞는다.

- 안정성이 필요한 작업: 조사 → 보고 → 승인 → 수정 → 검증
- 단순 작업: 짧은 계획 후 수정
- 긴 디버깅: 세션을 길게 유지하되, 중간에 WORK_LOG 요약을 남김
- 컨텍스트가 무거워진 작업: 압축 또는 새 세션으로 이관
- 도구 변경 시: 이 문서 3개와 해당 adapter 문서를 새 폴더에 이식

## 6. 도구별 adapter 문서

도구별 세부 설정은 부수 문서로 둔다.

```text
/adapters/codex/CODEX_SETUP_AND_OPERATION.md
/adapters/cursor/CURSOR_SETUP_AND_OPERATION.md
/adapters/antigravity/ANTIGRAVITY_HARNESS_AND_OPERATION.md
```

도구별 문서는 설치·세팅·권한·운용법만 담는다.  
프로젝트 작동 원리나 제품 설계는 반복하지 않는다.  
그 내용은 반드시 `나눔피아노_전체_설계.md`로 통합한다.

## 7. 작업 중단 후 빈 폴더 복구 절차

작업이 중단되거나 새 도구로 옮겨야 할 때는 아래 절차를 따른다.

### 7-1. 빈 폴더에 중요 문서 이식

빈 폴더에 먼저 아래 3개를 복사한다.

```text
CONSTITUTION.md
WORK_LOG.md
나눔피아노_전체_설계.md
```

필요한 도구가 정해졌다면 해당 adapter 문서도 복사한다.

예:

```text
/adapters/codex/CODEX_SETUP_AND_OPERATION.md
```

### 7-2. 기초 설치

도구별 adapter 문서를 보고 최소 설정만 한다.

- Codex면 Codex adapter
- Cursor면 Cursor adapter
- Antigravity면 Antigravity adapter

### 7-3. 작업자에게 첫 지시

첫 지시는 반드시 아래 구조로 한다.

```text
이 프로젝트는 새로 시작하는 것이 아니다.
CONSTITUTION.md, WORK_LOG.md, 나눔피아노_전체_설계.md를 먼저 읽어라.
현재 상태를 요약하고, 지금 이어받아야 할 작업과 금지 사항을 보고하라.
승인 전에는 파일을 수정하지 마라.
```

### 7-4. 작업 재개

작업자가 보고하면 사용자가 승인한다.  
그 다음에만 수정한다.

---

## 8. Codex, Cursor, Antigravity 공통 통제

모든 작업자는 아래 순서를 따른다.

```text
조사
→ 보고
→ 사용자 승인
→ 수정
→ diff 확인
→ 실행 또는 검증
→ WORK_LOG 맨 위 기록
```

Full Access, 자동 실행, 무제한 수정, 자동 배포는 기본값으로 쓰지 않는다.  
필요할 때만 제한적으로 사용하고, 이유를 WORK_LOG에 남긴다.

## 9. 완료 정의

작업 완료는 아래를 모두 만족해야 한다.

1. 사용자가 요청한 범위 안에서 끝났다.
2. 변경 파일이 명확하다.
3. diff 또는 변경 요약이 있다.
4. 검증 방법과 결과가 있다.
5. 실패가 있으면 실패 원인과 수정 방법이 있다.
6. 남은 문제가 적혀 있다.
7. WORK_LOG 맨 위에 기록이 추가되었다.
8. 필요 시 Git 상태 또는 커밋 해시가 있다.

위 조건이 없으면 완료가 아니다.

## 10. 최종 선언

이 문서는 멋있는 설명문이 아니다.  
작업자가 바뀌어도 프로젝트를 살리기 위한 복구 문서다.

작업자는 이 문서를 읽고 아래를 즉시 복구해야 한다.

- 프로젝트 정체
- 현재 진행 상태
- 확정된 원리
- 다음 작업
- 금지 사항
- 검증 방식
- 도구 운용 방식

복구할 수 없다면 문서가 잘못된 것이다.
