## 0. 목적

이 문서는 Antigravity 또는 유사 하네스형 작업 환경에서 나눔피아노 프로젝트를 이어받기 위한 adapter 문서다.  
프로젝트 작동 원리는 반복하지 않는다. 먼저 아래 문서를 읽게 한다.

```text
CONSTITUTION.md
WORK_LOG.md
나눔피아노_전체_설계.md
```

---

## 1. 하네스 자동 적용 영역

사용자 제공 하네스 가이드 기준으로 Antigravity 계열에서는 아래 구조가 중요하다.

```text
agent/rules
agent/skills
agent/workflows
.antigravityignore
```

역할:

- `agent/rules`: 프로젝트 전용 행동 지침
- `agent/skills`: 도메인 전문 지식
- `agent/workflows`: 반복 작업 자동화
- `.antigravityignore`: 읽지 않아도 되는 대용량 폴더, 로그, 빌드 산출물 제외

주의:

- 자동 적용을 맹신하지 않는다.
- 작업 시작 시 “규칙과 스킬을 읽었는지” 보고하게 한다.
- 하네스 설정은 보조 장치이며, 최종 기준은 3대 문서다.

---

## 2. 사용자 직접 지시가 필요한 영역

아래는 사용자가 팀장처럼 직접 지시해야 한다.

- 모델 선택
- 계획 먼저 여부
- 브라우저 검수 실행 여부
- 병렬 에이전트 사용 여부
- MCP 서버 사용 여부
- 배포 여부
- 삭제 또는 이동 여부
- 대규모 리팩터링 여부

## 3. 새 폴더에서 복구 절차

### 3-1. 빈 폴더에 이식할 파일

```text
CONSTITUTION.md
WORK_LOG.md
나눔피아노_전체_설계.md
/adapters/antigravity/ANTIGRAVITY_HARNESS_AND_OPERATION.md
```

필요 시:

```text
agent/rules/
agent/skills/
agent/workflows/
.antigravityignore
```

### 3-2. Antigravity 첫 지시문

```text
이 프로젝트는 새로 시작하는 것이 아니다.

CONSTITUTION.md, WORK_LOG.md, 나눔피아노_전체_설계.md를 먼저 읽어라.
그 다음 agent/rules, agent/skills, agent/workflows, .antigravityignore가 있는지 확인하라.
현재 상태와 이어받을 작업을 보고하라.
승인 전에는 파일 수정, 삭제, 이동, 커밋, 배포를 하지 마라.
```

## 4. 디자인 자동화 원칙

사용자 제공 가이드 기준으로 디자인 흐름은 아래처럼 둘 수 있다.

- Stitch: 시각적 시안과 아이디어
- v0.dev: 디자인을 웹 코드로 변환
- Antigravity: 생성 결과물을 통합하고 고도화

단, 이 흐름은 고정 법칙이 아니라 도구 조합이다.  
현재 프로젝트의 작동 원리를 해치면 사용하지 않는다.

---

## 5. 브라우저 검수 원칙

UI 작업 후에는 가능한 경우 브라우저 검수를 수행한다.

검수 항목:

- 로컬 서버 실행
- 데스크톱 화면
- 모바일 화면
- 클릭 흐름
- 스크롤
- 링크 오류
- 가로 오버플로
- 콘솔 에러
- 주요 스크린샷 저장

검증 없는 UI 완료 선언은 금지한다.

---

## 6. 병렬 에이전트 원칙

병렬 작업은 멋으로 쓰지 않는다.  
아래 조건에서만 사용한다.

- 역할이 분명할 때
- 서로 파일 충돌이 없을 때
- 검증 기준이 정해져 있을 때
- 결과를 한 사람이 통합 검수할 때

예:

- 한 에이전트: UI 검수
- 한 에이전트: 문서 중복 조사
- 한 에이전트: 자산 경로 확인

금지:

- 여러 에이전트가 같은 파일을 동시에 수정
- 통합 검수 없이 결과 병합
- WORK_LOG 없이 완료 선언

---

## 7. MCP 사용 원칙

MCP는 유행이 아니라 실제 필요가 있을 때만 쓴다.

사용 기준:

- 시간을 줄이는가
- 복붙을 줄이는가
- 검증을 강화하는가
- 권한 범위를 통제할 수 있는가

MCP 서버 추가 전에는 목적과 권한을 WORK_LOG에 남긴다.
